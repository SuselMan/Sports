# Architecture & Stack Review

## Overall Assessment
Стек хорошо подобран для pet-проекта: full-stack TypeScript, shared types, offline-first. Проект уже работает в проде. Ниже — что стоит рассмотреть перед "настоящим" продакшеном.

---

## Frontend

### Что хорошо
- **Offline-first** с IndexedDB — серьёзное конкурентное преимущество
- **PWA** — установка на телефон без App Store
- **Shared types** — единый источник правды для типов
- **Zustand** — лёгкий и предсказуемый стейт-менеджер
- **i18n на 10 языков** — готовность к международной аудитории
- **Custom UiKit** — контроль над UI, нет зависимости от тяжёлых библиотек

### Что стоит пересмотреть

#### 1. MUI + Custom UiKit = двойная зависимость
**Проблема**: Используются одновременно MUI (DatePicker, иконки) и свой UiKit. Это увеличивает bundle size и создаёт стилистическую несогласованность.
**Рекомендация**: Определиться с одним подходом. Варианты:
- Заменить MUI DatePicker на нативный `<input type="date">` (уже используется для мобильных) + свой десктоп-пикер в UiKit
- Или убрать UiKit и перейти полностью на MUI
- **Моя рекомендация**: Оставить UiKit как основу, заменить MUI-зависимости кастомными или лёгкими альтернативами. Это соответствует философии минимализма.

#### 2. Bundle size
**Проблема**: lodash, recharts, MUI, dayjs — каждая библиотека добавляет вес.
**Рекомендация**:
- Убедиться что lodash импортируется по функциям (`lodash/debounce`), а не целиком
- Рассмотреть `lightweight-charts` вместо recharts (если нужны только line charts)
- Проверить tree-shaking MUI

#### 3. Нет error boundary
**Проблема**: Ошибка в одном компоненте роняет всё приложение.
**Рекомендация**: Добавить React Error Boundary на уровне pages.

## Backend

### Что хорошо
- Простая и понятная архитектура
- Row-level security через userId
- Soft deletes для sync
- Валидация на уровне Mongoose schemas

### Что стоит пересмотреть

#### 1. Нет service layer
**Проблема**: Вся бизнес-логика в route handlers. Пока проект маленький — это ОК. При росте станет сложно тестировать и переиспользовать.
**Рекомендация**: Пока не менять. Рефакторить в services только когда появятся тесты или дублирование логики.

#### 2. Express 4 → Express 5 или Fastify?
**Проблема**: Express 4 не поддерживает async error handling из коробки. Каждый handler обёрнут в try/catch.
**Рекомендация**: Пока не менять. Express 4 работает, миграция не стоит затрат. Если захочется — express-async-errors решает проблему одной строкой.

#### 3. Нет rate limiting
**Проблема**: API открыт для abuse — можно заDDoSить или забить базу мусором.
**Рекомендация**: Добавить `express-rate-limit` — 5 строк кода, критично для прода.

#### 4. Нет request logging
**Проблема**: Сложно диагностировать проблемы в проде.
**Рекомендация**: Добавить `morgan` или `pino` — минимальные усилия, большая польза.

#### 5. CORS_ORIGIN = '*'
**Проблема**: Любой сайт может делать запросы к API.
**Рекомендация**: В проде установить `CORS_ORIGIN=https://kektrack.online`.

## Database

### Что хорошо
- MongoDB хорошо подходит для гибких схем и per-user данных
- Правильные индексы (userId + archived + date)

### Что стоит пересмотреть

#### 1. Нет миграций
**Проблема**: При изменении схемы нет автоматического способа обновить данные.
**Рекомендация**: Добавить `migrate-mongo` — простая система миграций для MongoDB.

#### 2. Нет backup стратегии
**Проблема**: Потеря данных = потеря пользователей.
**Рекомендация**: Настроить `mongodump` по крону или использовать MongoDB Atlas с автобэкапами.

## Infrastructure

### Что хорошо
- GitHub Actions CI/CD — автодеплой
- PM2 — автоперезапуск при крашах
- Let's Encrypt — бесплатный SSL

### Что стоит пересмотреть

#### 1. SSH с паролем в CI
**Проблема**: `SSH_PASS` в secrets менее безопасен чем SSH key.
**Рекомендация**: Перейти на SSH key authentication.

#### 2. Нет Docker
**Проблема**: "Works on my machine" проблемы, сложнее воспроизвести прод-окружение.
**Рекомендация**: Docker не обязателен для одного VPS. Но docker-compose для локальной разработки (MongoDB) был бы полезен.

#### 3. Нет health monitoring
**Проблема**: Не узнаешь о проблемах пока пользователь не пожалуется.
**Рекомендация**: Бесплатный uptime monitoring (UptimeRobot, Better Stack) + PM2 monitoring.

#### 4. CI не запускает lint/tests
**Проблема**: ci.yml только делает build, не проверяет качество кода.
**Рекомендация**: Добавить lint в CI pipeline.

## Security

### Критичные проблемы
1. **CORS `*`** — ограничить доменом
2. **Нет rate limiting** — можно забить API
3. **JWT 30 дней без refresh** — при компрометации токена нет способа отозвать доступ

### Рекомендации
- Добавить rate limiting (express-rate-limit)
- Ограничить CORS
- Рассмотреть refresh token flow (но для MVP не критично)
- Добавить helmet.js для security headers
